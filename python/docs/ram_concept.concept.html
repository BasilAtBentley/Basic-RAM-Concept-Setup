ram_concept.concept module¶

class Concept(url)¶

Bases: object

Concept represents the RAM Concept process that hosts the data (“file”).

A maximum of 1 Model may be managed by a Concept process at anytime. However, there can be multiple Concept processes active (which indirectly allows multiple Models to be active).

This class should not be subclassed.

Parameters:

url (str) –

static attach_to_concept(port)¶

Attaches to an existing RAM Concept that will serve this API.

The caller is responsible for having started the Concept process listening to the given port.

Concept should have been started with the command:

concept.exe -apiServer -port <port_number> -inactivityTimeout <timeout_in_seconds> -version <api_version>

port_number is in the 1..65535 range; api_version has the format 8.2.1

This method is not commonly used.

Parameters:

port (int) – The port number for the server to use

Returns:

A Concept object, ready for use.

Return type:

Concept

static start_concept(headless=True, path=None, port=None, start_timeout_seconds=None, inactivity_timeout_seconds=None, log_file_path=None)¶

Start the RAM Concept server that will serve this API.

Parameters:

  • headless (bool) – true to run the server headlessly, false to run the server with a GUI

  • path (str | None) – path to the RAM Concept exe to start (if None, default location will be used)

  • port (int | None) – The port number for the server to use (if None, an available port will be used)

  • start_timeout_seconds (int | None) – The number of seconds to allow the server to start up before timing out (if None, a default value is used)

  • inactivity_timeout_seconds (int | None) – The number of seconds that the server will wait for the next API command before timing out (if None, a default value is used)

  • log_file_path (str | None) – Path to a log file that is written to by the RAM Concept.exe (rarely used)

Returns:

A Concept object, ready for use.

Return type:

Concept
